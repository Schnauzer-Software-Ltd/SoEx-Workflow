A · Onboarding

Start, then deliver the events the saga waits on (IDP & invitee callbacks).
Start onboarding → look up the user, create the account, then wait. Account verified → reserve a seat, send the invite, then wait. Invite accepted → assign the seat, complete & shred the key. (If a wait times out the saga compensates — release the seat / abandon.)

B · Subscription renewal

Force a decline to drive the dunning wait, then "fix the card" with a payment update.
Force decline (period 1) first so the charge fails (optional). Start renewal → charge the period; on success, invoice and continue-as-new into the next period (up to 3) then complete; on a decline, enter dunning and wait. Payment updated → retry the charge and carry on. After the max attempts it cancels.

Not wired on this host — its runtime/mode doesn't drive subscription renewal. Exercise it on the InProc, Temporal or DurableTask host.

C · Offboarding

Native fan-out: one governed revocation per downstream system, in parallel.
Start offboarding → fan out in parallel, one governed revocation per downstream system (mail, VPN, billing portal, wiki), then archive the must-retain record and shred the key. No further buttons — the fan-out completes on its own.
Offboarding is a native parallel fan-out, so it needs a runtime that can fan out. Not available on the portable InProc shell or the onboarding-only Elsa host — try Temporal, DurableTask or Restate.

D · Right to erasure

Sweep every in-flight instance for a subject — crypto-shred keys, retain the carve-out.
Forget subject → find every in-flight instance still indexed for the subject, run each one's terminal, crypto-shred its per-instance key (its journal becomes unrecoverable), prune the subject index, and write the must-retain carve-out outward. Run a couple of flows first so there's something to erase.

Durability

Restart the host mid-flow (over the same persistent store), then resume — the saga survives the process going away.
Onboarding's Start onboarding parks the saga on the invite-accepted bookmark (persisted to SQLite) → Restart host disposes the process and rebuilds a fresh one over the same database → Onboarding's Invite accepted → it resumes on the new process and completes. Durable state outlives the process.
Interactive host-restart durability is specific to the Elsa host (it persists to SQLite and resumes on a fresh process). Other runtimes hold durable state in their server or scheduler.
